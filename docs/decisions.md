# Architecture Decisions

Lightweight log of decisions worth remembering the reasoning behind, not just the outcome.

---

## 1. The Investment Council stays mocked data in `prisma/seed.ts` through Sprint 1

**Status:** Accepted — Sprint 1 close-out.

**Context**

The original Sprint 1 outline called for the Investment Council to exist as a modular
service architecture: nine typed `CouncilMember` modules plus a `ChiefInvestmentOfficer`
orchestrator that consumes their outputs and produces the Brief. `src/services/council/`
was scaffolded for this in Task 1.

That architecture was never built. Instead, `prisma/seed.ts` contains nine realistic,
hand-written Council assessments as static data, inserted directly.

**Decision**

This is intentional, not incomplete. Through Sprint 1 it became clear that the higher-value
work was validating the reading experience — the Brief, the Dashboard, the design language,
and the domain model underneath them — rather than building a service architecture whose
only job, for now, would be moving static content from one file to another.

A `CouncilMember` interface and orchestrator with no real per-executive logic behind it
would be structure without responsibility: modularity for its own sake, which the project's
own engineering principles (Constitution: "Favor modularity over optimization") argue
against when there's nothing yet to modularize.

**When this changes**

The Council service layer gets built in the sprint where individual members begin
consuming real inputs — market data, fundamentals, technicals — and producing independent,
non-hardcoded assessments. At that point each module has genuine responsibility, and the
`CouncilMember` interface defined back in the architecture review (role, confidence,
rationale, risks, change trigger) becomes the real contract implementations are built
against, rather than a shape `seed.ts` happens to already match.

Until then, `getLatestBrief()` / `getLatestBriefDetail()` in `src/lib/data/brief.ts` read
directly from whatever is in the `briefs` table — seeded today, generated by the real
Council later — without the Dashboard or Brief page needing to change at all. That's the
part of this decision worth relying on: the persistence layer and the UI were built against
the _shape_ of a Brief, not against `seed.ts` itself, so swapping mocked generation for real
generation is additive, not a rewrite.

## 2. Rule B (Concentration) cites the concentration threshold, not a structural Risk link

**Status:** Accepted — Sprint 2 implementation.

**Context**

The Sprint 2 design doc specified Rule B as firing when "a single holding ≥8% of portfolio
_and_ referenced by a Risk's synthesis." `Risk` has no relation to `Company` in the schema —
only to the `CouncilAssessment`s that raised it. The seeded "Mega-Cap AI Concentration" risk
mentions NVDA-style names only in free-text description, not a foreign key.

**Decision**

Rather than pattern-matching a company name against Risk description text — a fragile
heuristic that would make "evidence" mean something different in this one rule than
everywhere else — Rule B fires on the concentration threshold alone. Its evidence citation
references the concentration guideline in today's Recommended Actions instead (`"Trim any
single position that exceeds 8% of total equity exposure"` — a real row, not a text match).
This keeps every rule's evidence a structural citation to an actual Brief field, consistent
with the "why am I seeing this?" requirement, rather than fabricating a link the schema
doesn't have.

**When this changes**

If a future sprint adds a real `Risk` ↔ `Company` relation (e.g., Risks generated by a real
Council citing specific holdings), Rule B should switch to citing the matching Risk directly
alongside — or instead of — the Recommended Actions citation it uses today.

## 3. The Recommendation Engine's target architecture is documented, not built

**Status:** Accepted — pre-Sprint 3.

**Context**

A full architecture review proposed evolving `src/lib/portfolio/rules.ts` from four
standalone functions into a formal subsystem: a registered rule contract, a separate
ranking step, structured (not string) evidence citations, and an `InvestorProfile`
input the whole engine takes. The review also established two things worth treating as
settled regardless of how the rest is sequenced:

1. **The explainability principle.** Every recommendation's evidence must resolve to a
   real, structured reference — a specific `Risk`, `Opportunity`, `AllocationTarget`, or
   `RecommendedAction` row — never a generated or fabricated sentence. Decision #2 above
   is the concrete precedent this principle already produced.
2. **The AI boundary.** A statistical or learned model may eventually decide which rules
   fire or how they're weighted (ranking). It may never generate the evidence citation
   itself. AI enters at ranking, not at rule-authoring.

**Decision**

The registry, the ranking subsystem, and `InvestorProfile` are **target-state, not
current-sprint** work. At four rules, `computeRecommendedChanges`'s current shape — a
flat list of function calls, hardcoded severity integers — is still the right tradeoff.
Building a registry now would repeat the exact mistake decision #1 already named and
avoided: structure without responsibility, modularity for code that doesn't yet exist.

The Recommendation contract, the explainability principle, and the AI boundary are kept
as binding design constraints starting now, even though the registry that would enforce
them structurally isn't built yet. Concretely: new rules added before the registry
exists should still return evidence as a real citation, in the spirit of the eventual
contract, even while `rules.ts` stays a flat file.

**When this changes**

The registry gets built when a fifth rule category — not a fifth rule, a fifth _kind_ of
thing rules need to reason about — makes the flat-function shape genuinely awkward to
extend. "This is new since yesterday" (the direction Sprint 3 is heading) is the most
likely trigger: comparing today's fired recommendations against a prior day's introduces
a cross-cutting concern that doesn't fit cleanly into any single rule function, which is
exactly the kind of pressure that justifies the registry/ranking split. Until that
pressure is real, `rules.ts` stays exactly as legible as it is today.
